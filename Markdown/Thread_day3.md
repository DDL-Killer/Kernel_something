# 线程池

## 为什么需要线程池?

在c语言中,pthread_create()是一个昂贵的系统调用,涉及内核态切换 分配独立的线程栈

### 内核态切换 从`Ring3`坠入`Ring0`

* 用户态(Ring3):不能直接访问物理内存,不能修改页表,不能操作中断控制器
* 内核态(Ring0):操作系统内核

调用`pthread_create()`时

1. 软件中断
2. 特权级检查:CPU检查当前段寄存器的CPL,发现3
3. 寄存器上下文强保存:将当前进程所有的通用寄存器压入内核栈
4. 切换栈指针:`RSP`必须从用户空间的虚拟地址栈,切换到一个内核空间预分配的 极小且固定的内核栈(4KB 8KB)
5. 切换页表基址(CR3寄存器):线程共享进程地址空间,但内核代码位于高地址区域,MMU需要刷新TLB(Translation Lookaside Buffer,旁路转换缓冲)缓存
6. 执行内核代码:CPU开始执行`sys_clone`

### 分配独立的线程栈空间

`pthread_create`行为:直接调用`mmap`系统调用

此时:

1. `mmap`系统调用:内核需要为这个新线程寻找一段 **连续的 未使用的虚拟地址空间**
2. 创建VMA结构体:内核维护一个红黑树`vm_area_struct`,记录这段地址的权限(栈保护机制的物理基础)
3. 设置Guard Page(保护页):内核在栈底故意标记一页为`PROT_NONE`

### 调度器负载与缓存污染

* CFS调度器的唤醒抢占:完全公平调度器会计算`vruntime`,会立即抢占当前正在运行的线程,造成无意义上下文切换
* TLB刷退:如果新线程被调度到另一个CPU核心,页表共享,TLB快表是空的,启动执行时会导致一次`TLB MISS`和`Page Walk`

### 线程池的逻辑

1. 预先招人:启动时就创建固定数量的线程
2. 待命休息:没有任务时通过`pthread_cond_wait`进入睡眠,不占CPU
3. 任务队列:主线程将任务丢进链表队列,并`pthread_cond_signal`
4. 抢活干:线程唤醒后,去队列领取任务执行

## 核心的数据结构

```c
// 1. 任务结构体：就像一张工单
typedef struct task {
    void (*function)(void* arg); // 函数指针，指向要执行的任务
    void* arg;                   // 任务函数的参数
    struct task* next;           // 指向下一个任务（链表结构）
} task_t;

// 2. 线程池结构体：就像一个办公室
typedef struct {
    pthread_mutex_t lock;       // 互斥锁，保护任务队列
    pthread_cond_t notify;      // 条件变量，用于唤醒睡眠的线程
    pthread_t *threads;         // 线程句柄数组
    task_t *queue_head;         // 任务队列头
    int thread_count;           // 线程池里的员工数
    int queue_size;             // 当前排队的任务数
    int shutdown;               // 关门标志位（0:正常运行, 1:准备关门）
} threadpool_t;
```

## pthread_cond_init()

POSIX线程库中用于初始化条件变量的函数

* 条件变量是一种同步机制,它允许线程在某个特定条件满足前挂起等待,并由其他线程在满足条件时唤醒

```c
#include <pthread.h>

int pthread_cond_init(pthread_cond_t *cond,const pthread_condattr_t *attr);
```

### 参数说明

- **`cond`**：指向 `pthread_cond_t` 类型条件变量的指针，该变量将被初始化。
- **`attr`**：指向 `pthread_condattr_t` 类型属性对象的指针，用于设置条件变量的属性。通常传入 **`NULL`**，表示使用系统默认属性

### 使用方法

条件变量有两种常见的初始化方式：

1. **动态初始化**

   在运行时通过函数调用进行初始化，可以对属性进行更精细的控制（如设置进程间共享）：

   ```c
   pthread_cond_t cond;
   // 使用默认属性初始化
   pthread_cond_init(&cond, NULL);
   ```

   

2. **静态初始化**

   在编译时使用宏 `PTHREAD_COND_INITIALIZER` 进行初始化，效果等同于动态初始化时 `attr` 参数为 `NULL`，但不进行错误检查，更为简洁 ：

   ```c
   pthread_cond_t cond = PTHREAD_COND_INITIALIZER;
   ```

### 返回值与错误处理

`pthread_cond_init` 成功时返回 **`0`**。失败时会返回以下错误码之一 ：

| 错误码       | 含义                                           |
| ------------ | ---------------------------------------------- |
| **`EINVAL`** | 参数 `attr` 指定的属性值无效。                 |
| **`ENOMEM`** | 系统内存不足，无法初始化条件变量。             |
| **`EAGAIN`** | 系统临时缺乏必要的资源（内存除外）。           |
| **`EBUSY`**  | 试图重新初始化一个已初始化但未销毁的条件变量。 |

## pthread_cond_wait()

条件变量机制中的核心阻塞函数,在底层执行了原子操作来防止唤醒丢失,将互斥锁的释放与线程睡眠合并为一个原子操作

```c
#include <pthread.h>

int pthread_cond_wait(pthread_cond_t *cond, pthread_mutex_t *mutex);
```

* **`cond`**: 指向已初始化的条件变量。

* **`mutex`**: 指向**已锁定**的互斥锁。

### 返回值与错误

| 返回值       | 含义                                                   |
| ------------ | ------------------------------------------------------ |
| **0**        | 成功唤醒并重新获得锁。                                 |
| **`EINVAL`** | 条件变量或互斥锁未初始化，或互斥锁不属于调用线程       |
| **`EPERM`**  | 互斥锁当前**未被**调用线程锁定（这是最常见的使用错误） |

## 为什么需要队列锁

是一种**将竞争线程排成队列、让各线程在各自独立存储单元上自旋的高级锁机制**。它的出现，主要是为了解决传统自旋锁在多核 CPU 环境下存在的几个严重的底层性能瓶颈。

### 传统自旋锁的代价：总线风暴与不公平

最简单的自旋锁（TASLock，Test-And-Set Lock）使用一个全局的 `state` 变量表示锁状态。所有竞争线程都在一个 **while 循环** 中反复执行原子操作来抢夺这个变量。

从底层来看，每次原子写操作（如 `getAndSet` 或 `compare_and_swap`）都会触发 CPU 的**缓存一致性协议**（如 MESI）：

1. **独占总线**：该操作会锁定缓存行，并通过总线广播失效信号（Invalidate），强制所有其他 CPU 核心缓存中的该变量副本失效。
2. **缓存失效与重新加载**：其他核心发现自己的缓存失效，只能从内存或持有最新数据的核心处重新加载。
3. **总线流量风暴**：当锁释放时，所有正在自旋的线程几乎**同时**发起请求，导致总线流量瞬间达到峰值，造成严重的“惊群效应”。

此外，这种随机竞争完全**不公平**，一个倒霉的线程理论上可能永远抢不到锁（饥饿）

### 队列锁的原理：化整为零，排队叫号

队列锁的核心思想是**将“竞争同一个变量”转变为“排队等待前一个节点的通知”**。它将线程组织成一个显式或隐式的队列，每个线程在其**自己独立**的存储单元（节点）上自旋。

根据实现方式，主流的队列锁有以下几种形态：

| 锁类型             | 核心机制                                                     | 自旋位置                 | 优点                                                     | 缺点                                                     |
| ------------------ | ------------------------------------------------------------ | ------------------------ | -------------------------------------------------------- | -------------------------------------------------------- |
| **数组锁 (ALock)** | 所有线程共享一个数组，通过原子自增获取**槽位号**，每个线程盯着自己的 `flag[slot]` | 数组的特定元素           | 公平（FIFO），无饥饿；缓存一致性流量低                   | 空间开销大，需预先知道最大并发数                         |
| **CLH 队列锁**     | 形成一个隐式链表，每个线程持有**前驱节点**的引用，并在**前驱节点的状态域**上自旋 | 前驱节点的 `locked` 变量 | 空间效率高（O(L+n)），公平                               | 在 NUMA 架构（非一致内存访问）下，访问前驱的内存可能较慢 |
| **MCS 队列锁**     | 形成显式链表，每个线程在自己的**本地节点**状态域上自旋，前驱负责修改它的值 | 本地节点的 `locked` 变量 | 空间效率高，对 NUMA 架构更友好（只需在自己的内存上自旋） | 释放锁的逻辑较复杂，需处理链表尾部的情况                 |

### 解决两个核心问题

1. 消除总线风暴:每个线程在独立的缓存行上自旋,大幅降低了因缓存一致性协议导致的总线流量
2. 保证公平性:锁分配严格遵循先来后到,彻底避免线程饥饿

### 锁的正确认识

* 锁本身并不分配任务,锁只是保证任务队列(共享的数据结构)不被破坏

* 锁只在调度(分发)的瞬间生效,执行阶段是完全无锁的并行
